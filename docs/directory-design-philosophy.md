## 1. このプロジェクトのディレクトリ設計思想

### バックエンドの構造

```
backend/app/
├── auth/        ← 認証という「機能」
│   ├── router.py
│   ├── service.py
│   └── dependencies.py
├── memos/       ← メモという「機能」
│   ├── router.py
│   └── service.py
├── tags/        ← タグという「機能」
│   ├── router.py
│   └── service.py
├── models.py    ← 全モデルを1ファイルに集約
└── schemas.py   ← 全スキーマを1ファイルに集約
```

**選んだ理由: 機能ベース（Feature-based）構成**

「認証に関するものは `auth/` にまとまっている」という構造にしました。

```
# 機能ベース → 関連ファイルが近くにある
auth/router.py       ← 認証のエンドポイント
auth/service.py      ← 認証のロジック
auth/dependencies.py ← 認証の依存関係

# vs. レイヤーベース（よくある対比）
routers/auth.py
services/auth.py
dependencies/auth.py
```

「認証を修正したい」となったとき、機能ベースなら `auth/` フォルダを開くだけで完結します。
レイヤーベースだと `routers/`・`services/`・`dependencies/` と3つのフォルダを行き来することになります。

### フロントエンドの構造

```
frontend/app/
├── auth/login/
│   └── page.tsx      ← URL(/auth/login)がそのままファイルパスになる
├── memos/
│   ├── page.tsx      ← メインページ
│   └── components/   ← このページ専用のコンポーネント
│       ├── MemoPane.tsx
│       ├── MemoEditor.tsx
│       ├── SearchBar.tsx
│       ├── TagSelector.tsx
│       └── TagCreator.tsx
└── lib/
    └── api.ts        ← API通信をまとめた共通モジュール
```

**選んだ理由: Next.js App Router の規約に従う**

Next.js App Router はファイルシステムがそのままURLになる設計です。
`app/memos/page.tsx` → `/memos` にアクセスすると表示される、という規約に乗ることで
ルーティング設定を自分で書く必要がなくなります。

## 2. 一般的な設計思想の種類

### レイヤードアーキテクチャ（Layered Architecture）

```
プレゼンテーション層（Router / Controller）
        ↓
ビジネスロジック層（Service）
        ↓
データアクセス層（Repository / ORM）
        ↓
データベース
```

**核心思想:** 役割ごとに層を分ける。上の層は下の層だけを呼ぶ。

| | 内容 |
|--|------|
| メリット | 各層の責任が明確。テストしやすい |
| デメリット | 機能を1つ追加するとき、全層にファイルを追加しなければならない |
| 採用例 | Spring Boot（Java）、Rails（Ruby）、Django（Python） |

---

### MVC パターン（Model-View-Controller）

```
Model      → データとビジネスロジック（DBとのやり取り）
View       → 画面の表示
Controller → リクエストを受けて Model を呼び、View に渡す
```

**核心思想:** レイヤードアーキテクチャの一種。Web フレームワークで最も広く使われてきた。

| | 内容 |
|--|------|
| メリット | シンプルで理解しやすい |
| デメリット | アプリが大きくなると Controller や Model が肥大化しやすい（Fat Controller / Fat Model 問題） |
| 採用例 | Rails、Laravel（PHP）、Django |

---

### クリーンアーキテクチャ（Clean Architecture）

```
┌─────────────────────────────────┐
│  外部（DB・UI・API）             │ ← 詳細。いつでも差し替えられる
│  ┌─────────────────────────┐    │
│  │  インターフェース層      │    │ ← Controller・Repository の実装
│  │  ┌───────────────────┐  │    │
│  │  │  ユースケース層    │  │    │ ← アプリのビジネスルール
│  │  │  ┌─────────────┐  │  │    │
│  │  │  │  エンティティ  │  │  │    │ ← 中心。フレームワークに依存しない
│  │  │  └─────────────┘  │  │    │
│  │  └───────────────────┘  │    │
│  └─────────────────────────┘    │
└─────────────────────────────────┘
```

**核心思想:** 「ビジネスロジックをフレームワークや DB から独立させる」

依存の方向が常に内側（エンティティ）に向かう。
DB を PostgreSQL から MySQL に変えても、ビジネスロジックは一切変更しなくていい。

| | 内容 |
|--|------|
| メリット | 変更に強い。テストしやすい |
| デメリット | コード量が多い。小規模プロジェクトには過剰 |
| 採用例 | 大規模な企業システム、長期運用が前提のプロダクト |

---

### DDD（ドメイン駆動設計 / Domain-Driven Design）

**核心思想:** 「ビジネスの言葉でコードを書く」

```python
# DDD 的でない
def update_record(id, data): ...

# DDD 的
def publish_article(article_id, author): ...
def cancel_order(order_id, reason): ...
```

ビジネスの概念（ドメイン）をそのままコードに反映させる。
「注文」「在庫」「請求」といったビジネス用語がクラス名やメソッド名になる。
開発者と非エンジニアが同じ言葉で話せるようになる（ユビキタス言語）。

| | 内容 |
|--|------|
| メリット | ビジネスの変化にコードが追随しやすい |
| デメリット | 習得コストが高い。小さなプロジェクトには合わない |
| 採用例 | ECサイト・金融・物流など複雑なビジネスルールを持つシステム |

---

### フィーチャースライス（Vertical Slice Architecture）

```
feature/
├── create-memo/     ← メモ作成に必要なものが全部ここにある
│   ├── handler.py
│   ├── service.py
│   ├── schema.py
│   └── test.py
├── delete-memo/
└── search-memo/
```

**核心思想:** 「1つの機能（スライス）を縦に切る。層で横に切らない」

今回のプロジェクトの `auth/`・`memos/`・`tags/` による分割はこの思想に近い。

| | 内容 |
|--|------|
| メリット | 機能追加・削除が独立してできる。チームで分担しやすい |
| デメリット | 機能間で共通ロジックが重複しやすい |
| 採用例 | マイクロサービス、中〜大規模チーム開発 |

## 3. 今回のプロジェクトとの対応

| 設計思想 | 今回のプロジェクトでの対応 |
|---------|----------------------|
| レイヤードアーキテクチャ | `router`（プレゼンテーション層）→ `service`（ビジネスロジック層）→ SQLAlchemy（データ層）の3層構造を採用 |
| MVC | `router.py` が Controller、`models.py` が Model に相当 |
| フィーチャースライス | `auth/`・`memos/`・`tags/` という機能単位の分割 |
| クリーンアーキテクチャ | 部分的に取り入れている（`service.py` がビジネスロジックを持ち、`router` から分離） |
| DDD | 今回は小規模なので本格採用はしていないが、`create_memo()`・`list_memos()` といった命名はドメインの言葉に沿っている |

---

## 4. まとめ

今回の設計は **「小規模プロジェクトに適した現実的な折衷案」** です。

```
完全なクリーンアーキテクチャ
  → 過剰（コード量が3倍になる）

完全なレイヤーベース
  → 機能追加のたびに複数フォルダを行き来する

↓ 今回の選択

機能ベースの分割（auth/ memos/ tags/）
  + 軽量なレイヤー分離（router / service）
  = 学習用途・中小規模に最適なバランス
```

### 規模と設計思想の目安

| 規模 | 推奨される設計思想 |
|------|----------------|
| 個人・学習用（今回） | 機能ベース + 軽量レイヤー |
| 小〜中規模チーム | MVC / レイヤードアーキテクチャ |
| 中〜大規模・長期運用 | クリーンアーキテクチャ / DDD |
| マイクロサービス | フィーチャースライス + DDD |

規模が大きくなるにつれて「クリーンアーキテクチャ」や「DDD」が選ばれるようになります。
Stage 3・4 で扱う AWS 移行・Terraform・Kubernetes の文脈では、これらの思想がより重要になってきます。
